# v0.1.5 Candidate Scope

This document narrows the likely `v0.1.5` iteration scope for `spring-correctness-linter`.

The goal is to keep the next release small, explainable, and low-noise. The repository already
expanded its proxy-boundary coverage substantially in `v0.1.4`, so `v0.1.5` should prefer
accuracy refinement, regression hardening, and release-pipeline stability over adding another
large batch of new rules.

## Release Theme

`v0.1.5` should be treated as an **accuracy and stability iteration**.

That means:

- tighten a few remaining high-confidence false-negative or false-positive boundaries
- strengthen regression coverage where recent changes proved easy to regress
- keep release and sample metadata aligned so tag-driven releases stay reproducible

## Primary Candidate Areas

### 1. `SPRING_ENDPOINT_SECURITY` high-confidence refinements

Keep this rule focused on explicit, local, and testable security intent.

Good candidates:

- inherited or composed security intent that is already visible in source and can be matched without speculative config parsing
- centralized-security auto-detection refinements that rely on resolvable Spring types, not naming conventions
- focused regressions around inherited request mappings, composed annotations, or source-local security-chain implementations

Do not add:

- path-policy inference from `SecurityFilterChain` DSLs
- gateway or infrastructure policy modeling
- convention-heavy guesses that are not visible in source

### 2. `SPRING_CACHEABLE_KEY` explicit-signal refinement

Keep this rule centered on cache identity that can be justified from source.

Good candidates:

- additional explicit `KeyGenerator` / `CachingConfigurer` / `CacheConfig` patterns that can be resolved from source
- regression coverage around project-wide key-generator auto-detection and cache fingerprint invalidation
- small fixes that reduce false negatives without widening to project convention guessing

Do not add:

- heuristic project-wide cache conventions inferred from method names or cache names alone
- speculative `CacheResolver` or infrastructure modeling without a concrete, low-noise source signal

### 3. Report and plugin regression hardening

Recent iterations changed:

- generated `rules-reference.md`
- `rules-governance.json`
- lightweight JSON output
- release metadata consistency
- sample version alignment

`v0.1.5` should keep this surface stable by adding regression tests when:

- generated docs change structure
- plugin configuration behavior changes
- release metadata or sample version alignment becomes part of release prep

### 4. Release workflow reproducibility

`v0.1.4` showed that release failures can still come from metadata drift rather than from rule logic.

Good candidates:

- additional guardrails around version alignment between root and module `pom.xml` files
- documentation or automated checks that make tag-driven releases less brittle
- keeping local pre-release commands aligned with what GitHub Actions actually runs

## Secondary Candidates

These are acceptable only if the primary items stay small:

- narrow `EVENTS` refinements around explicit async or transaction-phase combinations
- small generated-doc or sample-guide consistency improvements
- CI diagnostics that make failure reasons easier to identify without browsing full logs

## Not In Scope

The following should stay out of `v0.1.5` unless a concrete user report justifies them:

- large new rule families
- speculative Spring Security configuration parsing
- broad cache-convention inference
- heavy external-rule plugin architecture changes beyond the current `ServiceLoader` SPI
- major reporting redesigns

## Suggested Exit Criteria

A `v0.1.5` candidate should satisfy all of these:

1. The release can be summarized in a short release note without introducing a new broad product theme.
2. Every behavior change has focused regression coverage in `linter-core` and, when relevant, in `linter-maven-plugin`.
3. Sample projects and generated docs still reflect the current built-in rule surface.
4. Release preparation remains reproducible through the documented Windows commands and the tag-driven GitHub workflow.

## Suggested First Pick

If only one engineering item is chosen next, prefer:

- one more high-confidence `SPRING_ENDPOINT_SECURITY` refinement, or
- one more explicit-signal `SPRING_CACHEABLE_KEY` refinement

Choose only one first, land regression coverage with it, then reassess noise before broadening scope.
